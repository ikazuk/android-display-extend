調査: Chrome 検索ボックスタップ時にキーボードが出ない

1. 現象

  • 外部ディスプレイの Chrome の検索ボックスをトラックパッドでタップ → IME が出ないことがある
  • フォーカスを一度外してもう一度タップすると出る
  • 毎回ではなく不安定

2. このアプリが IME を抑制しうるメカニズム（コードから特定）

仮説 A: タップ遅延 + setFocusedRootTask の競合 (最有力)

タップ操作のフロー:

  1. ユーザーがタッチパッドをタップ
  2. _scheduleTapReplay() で doubleTapTimeout (300ms) 遅延
  3. 300ms 後に _replayPendingTap() がタッチイベントを外部ディスプレイに注入
  4. Chrome の EditText がフォーカスを得て IME 表示をリクエスト

問題: accessibility パス (_dispatchPendingTapViaAccessibility) では、タッチ注入の直前に service.setFocus(displayId) を呼ぶ (line 589)。これは setFocusedRootTask() を呼び、ウィンドウフォーカスが一瞬リセットされる。

Chrome の EditText が既にフォーカスを持っている状態で再度タップすると:

  1. setFocusedRootTask() → システムが現在のフォーカスを一旦解除して再設定
  2. タッチイベント注入 → Chrome の EditText にフォーカス (既にある)
  3. しかしシステム側では「フォーカス変化なし」と判断 → IME 表示リクエストが発行されない

2回目にタップすると出る理由: 1回目で一度フォーカスが外れて別の場所がフォーカスを得た後に、2回目で「新しくフォーカスを取得した」と認識されて IME が出る。

仮説 B: マウスイベントとして注入される (inputManager パス)

inputManager パスの _replayPendingTap() (line 554-565) はバッファした MotionEvent をそのまま注入する。元イベントは _obtainMotionEventWithOffset() で作られるが、source は元のタッチイベントの source を引き継ぐ。

しかし _injectMouseEvent() (line 743-775) では明示的に InputDevice.SOURCE_MOUSE + TOOL_TYPE_MOUSE を設定している。マウスクリックでは Android は IME を自動表示しない (ユーザーがマウスで操作する場合はソフトキーボード不要と判断される)。

ただし _replayPendingTap() はバッファしたイベント（元が touchscreen source）をそのまま注入するので、このパスでは問題ない可能性もある。確認が必要。

仮説 C: FLAG_ALT_FOCUSABLE_IM の影響

touchpadOverlay は FLAG_NOT_FOCUSABLE | FLAG_ALT_FOCUSABLE_IM で作られている。FLAG_ALT_FOCUSABLE_IM はこのウィンドウが「IME を受け入れるか」を反転するフラグ。

通常: FLAG_NOT_FOCUSABLE → IME 非表示。FLAG_NOT_FOCUSABLE | FLAG_ALT_FOCUSABLE_IM → IME表示が「元に戻る」(表示される方向)。

このフラグの組み合わせが Display 0 上の IME レイヤリングに影響し、外部ディスプレイからの IME 表示リクエストと競合する可能性はあるが、直接の原因になるとは考えにくい。

4. 切り分けに必要なデバッグログ

以下のログを追加すれば、どの仮説が正しいか切り分けられる:

ログ 1: タップ注入のパスとタイミング

// _replayPendingTap() 内
State.log("[TAP] replay via " + (inputManager != null ? "inputManager" : "accessibility")
    + " events=" + gestureState.pendingTapEvents.size());

// _dispatchPendingTapViaAccessibility() 内の setFocus 直前
State.log("[TAP] setFocus before accessibility tap dispatch");

ログ 2: 注入イベントの source

// _replayPendingTap() の inputManager パス、ループ内
State.log("[TAP] inject source=0x" + Integer.toHexString(event.getSource())
    + " tool=" + event.getToolType(0));

ログ 3: IME 表示変化の追跡

既に追加済みの [IME] visible= ログが使える。タップ後に IME が表示されたかどうか。

5. 推奨する次のステップ

  1. 上記ログを追加してビルド
  2. 再現させて確認:
    • タップ時に [TAP] replay via inputManager が出るか accessibility が出るか
    • inputManager パスなら: 注入イベントの source が SOURCE_TOUCHSCREEN (0x1002) か SOURCE_MOUSE (0x2002) か
    • accessibility パスなら: [TAP] setFocus の後に IME が出ないケースを確認
  3. 仮説確定後の修正候補:
    • 仮説 A: タップ注入前の setFocus を、既にフォーカスがあるなら skip する
    • 仮説 B: 注入イベントの source を明示的に SOURCE_TOUCHSCREEN に設定する
