Bug Report: トラックパッド表示中にGBoardのアルファベットが入力できない

再現環境: Pixel 10 Pro, GBoard
症状: トラックパッドを出した状態で builtin ディスプレイに GBoard が出ると、アルファベットキーが押せない。Enter/Backspace は百発百中で反応。Space/ピリオドは低確率で反応。ドラッグでマウスカーソルが動く。
ステータス: 原因切り分けのためデバッグログ追加済み

1. 有力仮説: InputConnection の commitText 問題

GBoard のアルファベットキーとファンクションキーで IME→アプリの配送メカニズムが異なる

GBoard は入力方法によって異なるメカニズムでテキストを送信します:

キーGBoard の送信方法状態
Enter InputConnection.sendKeyEvent(KEYCODE_ENTER) ✓ 動作
Backspace InputConnection.sendKeyEvent(KEYCODE_DEL) or deleteSurroundingText() ✓ 動作
アルファベット InputConnection.commitText("a", 1) ✗ 動作しない
Space commitText(" ", 1) or sendKeyEvent(KEYCODE_SPACE) (状況依存) △ 低確率で動作
ピリオド commitText(".", 1) or sendKeyEvent(KEYCODE_PERIOD) (状況依存) △ 低確率で動作

仮説: DISPLAY_IME_POLICY_FALLBACK_DISPLAY 設定下で、GBoard が Display 0 に表示されながら外部ディスプレイの EditText に入力する場合、sendKeyEvent() 経由の KeyEvent はクロスディスプレイで正しく配送されるが、commitText() / setComposingText() 経由のテキスト入力は InputConnection の接続先ディスプレイ問題により失敗する。

2. 対立仮説: タッチオーバーレイの矩形重複

排除できない可能性: touchpadOverlay と GBoard が矩形的に重なっていて、アルファベット領域へのタッチがオーバーレイに横取りされている可能性もまだ残っている。ただし、Backspace が 'm' キーの隣にあって百発百中なのは、この仮説と矛盾する。

もし矩形重複が原因なら、以下が当てはまるはず:

  • オーバーレイの下端が GBoard のアルファベット行と重なり、最下段 (Enter/BS) は外
  • しかし BS は 'm' の横にあるので、同じ Y 座標のはず → 矛盾

このことから、仮説 1 (InputConnection) の方が有力と判断。

3. 「ドラッグでカーソルが動く」の解釈

ユーザーが「GBoard のアルファベット部分を押すとカーソルが動く」と報告しているのは:

  • GBoard の上側（トラックパッド領域）に指が当たっている場合
  • GBoard のキーを押そうとして指がスワイプ的に動き、トラックパッド領域部分で ACTION_DOWN が発生
  • もしくは GBoard 表示時にトラックパッド領域をタッチした際の話

デバッグログにより、IME 表示中にオーバーレイにタッチが来ているかどうかが判明します。

4. デバッグログの使い方

追加したログと確認方法

タグ TouchpadActivity と TouchpadA11y でフィルタ:

adb logcat -s TouchpadActivity:D TouchpadA11y:D

ログ 1: IME 表示状態の変化

[IME-DEBUG] imeVisible=true imeHeight=732 touchpadArea=[1080x1520 @ y=120]
[IME-DEBUG] overlay screen pos=(0,120) size=1080x1520

確認ポイント: GBoard 表示時にオーバーレイのサイズが縮小されるか、元のまま残るか。

ログ 2: IME 表示中のオーバーレイへのタッチ

[IME-DEBUG] touch on overlay WHILE IME VISIBLE: action=DOWN local=(540,1200) screen=(540,1320) ...

確認ポイント:

  • ログが出る → オーバーレイがタッチを横取りしている（仮説2が正しい）
  • ログが出ない → GBoard がタッチを正常に受信している（仮説1が正しい）

ログ 3: KeyEvent の配送

[IME-DEBUG] onKeyEvent: keyCode=66 action=DOWN char=none source=0x101 displayId=0 device=Virtual

確認ポイント:

  • Enter (keyCode=66) / Backspace (keyCode=67) の KeyEvent が来る
  • アルファベットキー (keyCode=29-54) の KeyEvent が来ない

この結果が出れば、仮説1（commitText は KeyEvent を発生させないので onKeyEvent には来ない）が確定。

ログ 4: AccessibilityEvent (テキスト変化)

[IME-DEBUG] a11yEvent: type=0x10 pkg=com.example.app text=[hello] before=hell added=1 removed=0

確認ポイント:

  • TYPE_VIEW_TEXT_CHANGED (0x10) がアルファベット入力時に出ない → commitText が外部ディスプレイの EditText に到達していない
  • 出る → commitText は到達しているが別の問題で表示されていない

5. 判定フローチャート

GBoard のアルファベットキーを押す │ ├─ [IME-DEBUG] touch on overlay が出る? │ │ │ ├─ YES → オーバーレイがタッチ横取り (仮説2) │ │ 修正: オーバーレイをIME領域から外す │ │ │ └─ NO → GBoard がタッチを受信している → 次へ │ ├─ [IME-DEBUG] onKeyEvent でアルファベットの KeyEvent が来る? │ │ │ ├─ YES → KeyEvent は配送されている、別の問題 │ │ │ └─ NO → GBoard は commitText() を使っている → 次へ │ └─ [IME-DEBUG] a11yEvent TYPE_VIEW_TEXT_CHANGED が来る? │ ├─ YES → commitText は到達しているが描画されていない │ └─ NO → commitText がクロスディスプレイで失敗 (仮説1確定) → Android framework の InputMethodManagerService の問題 → アプリ側で対処: AccessibilityService で InputConnection を proxy するか、 IME policy を変更するか検討
